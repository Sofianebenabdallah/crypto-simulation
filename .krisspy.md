# Project Instructions

This is a **React + TypeScript + Tailwind CSS** project running on a **real Vite dev server** (custom stack — not Sandpack).

## Tech Stack

- **Framework**: React 18 with TypeScript
- **Bundler**: Vite (real dev server, not Sandpack)
- **Styling**: Tailwind CSS for utility-first styling
- **Icons**: Lucide React for beautiful, customizable icons
- **Routing**: React Router DOM + file-based routing (see below)
- **Images**: Unsplash for high-quality placeholder images

## Quick Start

### Using Icons
Import icons from lucide-react:
```tsx
import { Heart, Star, User, Settings } from 'lucide-react';

<Heart className="w-6 h-6 text-red-500" />
<Star size={24} color="gold" />
```

### Tailwind CSS
Use utility classes for styling:
```tsx
<div className="bg-blue-500 text-white p-4 rounded-lg shadow-md hover:bg-blue-600 transition-colors">
  <h2 className="text-xl font-bold mb-2">Card Title</h2>
  <p className="text-blue-100">Card description</p>
</div>
```

## Critical Rules

### Always Use Theme CSS Variables
When creating or modifying pages, ALWAYS use CSS variables from `/src/style/theme.css` instead of hardcoded Tailwind colors or values:
- Colors: `var(--primary)`, `var(--secondary)`, `var(--text-primary)`, `var(--text-secondary)`, `var(--bg-secondary)`, etc.
- Typography: `var(--font-size-base)`
- Shapes: `var(--border-radius)`
- Semantic: `var(--success)`, `var(--warning)`, `var(--danger)`

This ensures the Theme Editor can dynamically update all pages. Use inline styles with CSS variables, NOT Tailwind color classes like `text-purple-600` or `bg-indigo-500`.

### File-based Routing — NEVER edit src/App.tsx
Routes are auto-generated from files in `src/pages/**`. Just drop a file and the route appears:
- `src/pages/about.tsx` → `/about`
- `src/pages/blog/index.tsx` → `/blog`
- `src/pages/blog/[slug].tsx` → `/blog/:slug`

Files starting with `_` are skipped (helpers, layouts).

### Layouts use `children`, NOT `<Outlet />`
The optional root layout lives at `src/pages/_layout.tsx` and wraps every page. It MUST:
- Accept `{ children: React.ReactNode }` as props
- Render `{children}` where content should appear
- NEVER use `<Outlet />` from react-router-dom

```tsx
// CORRECT layout pattern
export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <div>
      <header>...</header>
      <main>{children}</main>
    </div>
  );
}
```

### Pages must NOT wrap themselves in a layout
The root layout is auto-applied by `src/App.tsx`. Pages MUST NOT import or render a layout component — this causes double wrapping.

```tsx
// WRONG — causes double layout
import RootLayout from './_layout';
const MyPage = () => <RootLayout><div>content</div></RootLayout>;

// CORRECT — just render content, App.tsx handles the layout
const MyPage = () => <div>content</div>;
```

### Update /MEMORY.md after each planning step
After completing each planning step (theme creation, design system update, page generation), the agent MUST update `/MEMORY.md` to reflect the current state:
- Record the chosen color palette, typography, and spacing tokens
- Mark completed steps with ✅
- Note any project-specific decisions or constraints

This keeps `/MEMORY.md` as a living document that future agents can read to understand what has already been done and why.
